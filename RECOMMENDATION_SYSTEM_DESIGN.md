# 推荐系统设计文档

## 1. 文档目的

本文档用于整理当前对推荐系统的整体设计共识，作为后续实现的基准说明。

本文档关注：

- 单一博主、长期狭窄输入场景下的推荐系统目标
- 全量语料构建、segment 保留原则、chunk 建立算法
- 错题驱动的用户画像思路
- 基于 word / chunk 索引的相关卡片推荐逻辑

本文档暂不关注：

- 具体数据库字段命名
- 具体 API 形态
- 前端交互细节
- 性能优化到最终形态


## 2. 项目前提

本项目的推荐系统建立在以下前提之上：

1. 系统长期围绕单一 YouTuber / 单一内容创作者构建。
2. 推荐的核心目标不是跨博主内容发现，而是帮助用户逐步适应该博主的语言系统。
3. 系统会尽可能下载该博主的全量视频，并将视频切分为大量 `segments`。
4. `segment` 是当前已经验证成功的核心训练单位。
5. 推荐系统不再依赖此前失败的 chunk 展示功能，而是重新基于博主全量语料建立 chunk 索引。


## 3. 总体目标

推荐系统需要回答两个实际问题：

1. 用户当前在哪些词、哪些表达、哪些重复语块上反复出错？
2. 当用户出错后，系统应该推荐哪些“相关但更有训练价值”的卡片继续练习？

因此，本系统的核心不是“抽象地给用户打标签”，而是：

- 把单一博主的全部语料组织成一个可检索的语言网络
- 把用户的错误挂接到这个网络上
- 再围绕错误点检索并推荐相关 segment


## 4. 设计原则

### 4.1 单一博主原则

所有推荐逻辑默认只在该博主语料内部进行，不以跨博主迁移为目标。

### 4.2 Segment 优先原则

训练、推荐、归因的最终落点都是 `segment`。  
`word` 和 `chunk` 是索引与归因单位，不是最终训练展示单位。

### 4.3 保留原始识别文本原则

segment 入库时必须保留 WhisperX 的原始识别文本，不能被后续修正覆盖。

推荐系统中建议区分三层文本：

- `raw spoken form`
  WhisperX 原始识别文本，作为事实层保留
- `display form`
  用于前端展示的文本，可做极轻微修正
- `index form`
  用于分词、chunk 建立、对齐、归因的规范化文本

### 4.4 Chunk 全自动原则

chunk 不做人工筛选。  
只要满足规则，就直接认定为 chunk。

### 4.5 错题驱动原则

用户画像的起点不是抽象标签，而是：

- 用户做错了哪些 `word`
- 用户做错了哪些 `chunk`
- 这些错误是否反复出现
- 这些错误是否被修复

### 4.6 最大区间原则

当一个错误 span 同时命中多个 chunk 时，统一选择“完全包含该错误 span 的最大 chunk”作为最终归因对象。


## 5. 全量语料处理流程

### 5.1 视频入库

系统按博主维度下载全量视频，并保留视频级元信息。

### 5.2 Segment 切分

每个视频经过：

1. 音频提取
2. WhisperX 转写与词级时间戳
3. 现有 segment 切分算法生成语义完整 segment

该流程沿用当前已经验证成功的 segment 主链路。

### 5.3 Segment 保存原则

每个 segment 至少需要具备以下概念层信息：

- 所属视频
- 原始识别文本
- 展示文本
- 规范化 token 序列
- 词级时间戳
- segment 在视频中的起止时间
- 五维难度信息

这里强调的是“信息层必须存在”，不代表数据库字段必须按此命名。


## 6. Chunk 建立算法

## 6.1 Chunk 定义

在单一博主全量语料中：

- 任意连续 token 序列
- 长度不少于 3 个 token
- 全语料中重复出现次数不少于 3 次

则直接认定为 chunk。

该定义不进行人工筛选，也不要求语言学上“优雅”。

## 6.2 规范化输入

chunk 建立一律基于 `index form` 进行。

建议规范化规则如下：

- 全部转小写
- 去掉纯标点
- 保留单词内部撇号
- 不跨 segment 建立 chunk
- 同一套规范化规则必须同时用于：
  - chunk 挖掘
  - word 索引
  - 用户答案对齐

## 6.3 建立过程

推荐采用“trigram 种子 + 向右扩展”算法：

1. 对每个 segment 的规范化 token 序列滑窗生成所有 trigram。
2. 统计全语料 trigram 出现次数。
3. 凡是出现次数不少于 3 的 trigram，直接认定为 chunk，并作为扩展种子。
4. 对每个种子观察其所有出现位置后面的下一个 token：
   - 若某个扩展后的更长片段出现次数仍不少于 3，则该更长片段也直接认定为 chunk
   - 继续迭代扩展，直到无法继续满足重复次数条件
5. 所有满足条件的重复片段全部保留，允许互相重叠。

## 6.4 Chunk occurrence

抽象 chunk 本体之外，还必须保留其在每个 segment 中的 occurrence 坐标。

每个 occurrence 至少需要表达：

- 属于哪个 segment
- chunk 文本
- 在 segment 规范化 token 序列中的起始词位置
- 在 segment 规范化 token 序列中的结束词位置
- 该 chunk 在全语料中的总出现次数

后续错误归因依赖的是 occurrence，不是只有 chunk 文本。


## 7. 两套核心索引

## 7.1 单词索引

建立 `word -> occurrences` 的倒排索引。

作用：

- 当错误无法归到 chunk 时，回退到 word error
- 基于单词错误推荐相关 segment

## 7.2 Chunk 索引

建立 `chunk -> occurrences` 的倒排索引。

作用：

- 将用户错误映射到重复语块
- 基于 chunk error 推荐相关 segment

## 7.3 Segment 局部索引

除全局倒排索引外，还需要支持：

- `segment -> [chunk occurrences]`

这是错误 span 映射到 chunk 时最直接需要的数据视图。


## 8. 用户错误归因算法

## 8.1 核心原则

错误归因永远在 reference 这边做，不在用户错误答案那边做。

原因：

- chunk 是从正确语料中建立的
- reference 具有稳定 token 坐标
- 用户答案本身不适合作为索引坐标系

## 8.2 归因流程

1. 取当前训练 card 对应的 reference segment。
2. 将 reference 与用户答案都转换为同一套规范化 token 序列。
3. 对两者做词级对齐，得到：
   - equal
   - substitute
   - delete
   - insert
4. 将连续错误位置聚合为最小错误 span。
5. 对每个最小错误 span，在当前 segment 的 chunk occurrences 中寻找所有“完全包含它”的 chunk。
6. 如果找到多个，按最大区间原则选择最大的 chunk。
7. 多个最小错误 span 若最终映射到同一个 chunk，则合并为一次 chunk error。
8. 若某个错误 span 找不到任何包含它的 chunk，则回退为 word error。

## 8.3 Span 定义

span 指 reference token 序列中的连续区间。

例如：

- `[3, 4]` 表示第 3 到第 4 个 token 构成的连续错误区间

插入错误不直接对应 reference token，可记为某个 token 边界，并使用“跨过该边界的最大 chunk”进行归因。

## 8.4 完全包含判定

假设：

- 错误 span 为 `[s, e]`
- 某个 chunk occurrence 为 `[c_start, c_end]`

则该 chunk 完全包含 span，当且仅当：

- `c_start <= s`
- `c_end >= e`

## 8.5 最大区间原则

当存在多个候选 chunk 完全包含同一错误 span 时，按以下顺序选择：

1. token 长度最长的 chunk
2. 若长度相同，选全语料频次更高的 chunk
3. 若仍相同，选起点更靠左的 chunk

## 8.6 归因产物

每次用户答错后，最终归因结果只能是以下二者之一：

- `chunk error`
- `word error`

这样可以保证推荐逻辑和用户画像逻辑简洁稳定。


## 9. 用户画像思路

## 9.1 画像本质

用户画像不是抽象人格，不是静态 level 标签，而是：

用户在该博主语言网络中的失败记录与修复轨迹。

## 9.2 用户画像的证据来源

用户画像由以下行为持续更新：

- 哪些 word 被判定为错误
- 哪些 chunk 被判定为错误
- 同一 word / chunk 的重复出错次数
- 某个错误点是否被后续练习修复
- 某个错误点修复花费了多少次训练
- 修复后是否在后续又重新出现

## 9.3 用户画像的第一版目标

第一版不追求复杂标签，而只追求回答：

- 用户当前最不稳定的高频错误词有哪些
- 用户当前最不稳定的高频错误 chunk 有哪些
- 哪些错误正在被修复
- 哪些错误具有高复发性

## 9.4 用户画像的优势

该画像方式有如下优点：

- 完全建立在可观察证据上
- 不需要一开始做复杂推断
- 能直接驱动推荐系统
- 随训练自然演化


## 10. 推荐系统逻辑

## 10.1 推荐触发

推荐主要由两类事件触发：

1. 用户在某个训练 card 上答错
2. 系统需要为用户生成下一批个性化训练卡片

## 10.2 以错误为中心的推荐

如果错误被归为 `chunk error`：

- 以该 chunk 为 key，从 chunk 倒排索引中检索所有相关 segment

如果错误被归为 `word error`：

- 以该 word 为 key，从 word 倒排索引中检索所有相关 segment

## 10.3 推荐结果的基本筛选原则

推荐候选 segment 建议满足以下约束：

- 仍来自同一博主
- 不优先重复推荐刚刚出错的同一 segment
- 难度尽量落在用户当前可承受区间附近
- 与当前错误点直接相关
- 在训练顺序上允许从更容易到更接近原题

## 10.4 推荐的实际目标

推荐的目标不是随机给“相似句子”，而是围绕当前错误点做局部强化：

- 对相同 word 的更多语境暴露
- 对相同 chunk 的更多语境暴露
- 用同一博主的重复表达帮助用户逐步建立自动化识别


## 11. 系统实现阶段建议

## 阶段 A：全量语料与索引基础

目标：

- 完成博主全量视频下载
- 完成稳定的 segment 入库
- 保留 raw / display / index 三层文本表示
- 建立 word 倒排索引

## 阶段 B：Chunk 挖掘

目标：

- 基于全量 segment 的规范化 token 序列建立 chunk
- 保存 chunk occurrences
- 建立 chunk 倒排索引

## 阶段 C：错误归因

目标：

- 在训练答题后，把错误稳定映射为 chunk error 或 word error
- 实现最大区间原则
- 保存错误事件

## 阶段 D：用户画像累积

目标：

- 按错误事件聚合用户的高频错误词与高频错误 chunk
- 跟踪错误修复与复发

## 阶段 E：推荐闭环

目标：

- 基于 word / chunk 错误检索相关 segment
- 输出下一批训练卡片
- 逐步形成“错题驱动推荐闭环”


## 12. 当前共识总结

本次讨论形成的关键共识如下：

1. 推荐系统服务于单一博主、长期狭窄输入场景。
2. 当前成功的 segment 主链路保留，不推翻。
3. WhisperX 原始识别文本必须保存，不允许被派生文本覆盖。
4. chunk 采用纯自动定义：
   - 连续 3 词及以上
   - 全语料重复出现至少 3 次
5. chunk 不做人工筛选，允许重叠。
6. 用户错误归因只分两类：
   - chunk error
   - word error
7. 错误归因采用最大区间原则：
   - 对错误 span 选取完全包含它的最大 chunk
   - 映射不上 chunk 时回退到 word
8. 用户画像以错题与修复轨迹为核心，而不是先验标签。
9. 推荐系统以 word / chunk 索引检索相关 segment，围绕错误点进行训练强化。


## 13. 后续待细化问题

后续实现前仍需进一步细化以下问题：

1. reference 与用户答案的具体词级对齐策略
2. 缩写、弱读、口语缩合的归一化规则
3. 推荐结果的排序策略
4. 同一错误点的去重与冷却机制
5. 错误被视为“已修复”的判定条件
6. 大规模全量视频下载与重跑时的增量更新策略
